# Initial Database Table List

This document outlines the minimum required tables for the platform's storage layer, specifying what they store, why they are necessary, and which modules interact with them.

---

### 1. sources
* **What it stores:** Metadata about data sources (e.g., API endpoints, Excel/CSV file paths, S3 bucket locations, database connection strings).
* **Why it is needed:** To manage, track, and parameterize data lineage. The system needs a centralized register to know where to fetch data from during ingestion.
* **Target Module:** `Integration & Ingestion Layer` (API Ingestion, File Processing, Streaming).

### 2. documents
* **What it stores:** Metadata of raw, unstructured files ingested into the system (e.g., PDF reports, Word files), including file name, file size, hash (for duplication check), and ingestion timestamp.
* **Why it is needed:** Acts as a landing log for unstructured data before it gets processed or split into chunks.
* **Target Module:** `Integration & Ingestion Layer` & `Processing & AI Layer` (Data Cleaning & Preprocessing).

### 3. document_chunks
* **What it stores:** Small text segments (chunks) extracted from `documents`, along with their corresponding vector embeddings (using `pgvector` data type).
* **Why it is needed:** Core component for semantic search. When a user asks a question, the RAG pipeline queries this table to find chunks with the highest cosine similarity.
* **Target Module:** `Processing & AI Layer` (Embedding Generation, RAG Pipeline).

### 4. structured_records
* **What it stores:** Cleaned, parsed, and highly structured transactional or operational data derived from files or APIs.
* **Why it is needed:** Serves as the primary source of truth for analytical queries and machine learning feature stores.
* **Target Module:** `Processing & AI Layer` (Feature Engineering, ML/DL Model Training) & `Application Layer` (Dashboards, Reports & Insights).

### 5. ingestion_logs
* **What it stores:** Details of each ingestion execution (e.g., log ID, source ID, start time, end time, status (SUCCESS/FAILED), total rows/files processed, error messages).
* **Why it is needed:** Essential for data quality and system monitoring. It helps data engineers troubleshoot failed API calls or corrupt files.
* **Target Module:** `Integration & Ingestion Layer` & Data Quality Monitoring.

### 6. pipeline_runs
* **What it stores:** Execution states of full ETL / ELT pipeline workflows (e.g., pipeline name, run ID, trigger type, duration, current state).
* **Why it is needed:** Orchestrates and tracks dependency health between multiple ingestion processes and data transformations.
* **Target Module:** `Integration & Ingestion Layer` (ETL / ELT Pipelines).

### 7. analytics_events
* **What it stores:** Logged events generated by system users or automated triggers (e.g., user interactions, feature usage, API latency metrics).
* **Why it is needed:** Aggregates operational and user metrics to feed business intelligence dashboards.
* **Target Module:** `Application Layer` (Dashboards, Reports & Insights).

### 8. rag_query_logs
* **What it stores:** History of user prompts submitted to the RAG Chatbot, the retrieved context chunks (IDs), and the final LLM-generated response.
* **Why it is needed:** Critical for monitoring chatbot performance, analyzing alignment/hallucinations, and calculating semantic drift over time.
* **Target Module:** `Application Layer` (RAG Chatbot).

### 9. prediction_logs
* **What it stores:** Inputs sent to ML/DL models and the corresponding output predictions, inference confidence scores, and model version IDs.
* **Why it is needed:** Facilitates continuous evaluation, auditing, and drift detection for model fine-tuning.
* **Target Module:** `Processing & AI Layer` (Model Tuning & Evaluation) & `Application Layer` (Generative AI Applications).
